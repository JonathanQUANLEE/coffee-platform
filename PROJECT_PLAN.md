# 咖啡热度平台 MVP 项目计划（Web + 微信小程序）

## 1. 项目一句话定位

做一个面向咖啡爱好者、咖啡新手、城市探店人群和精品咖啡商家的「咖啡馆发现 + 咖啡热度排名 + 咖啡知识百科 + 商家入驻」平台。

它不是单纯的信息展示网站，而是一个双端产品：

* **用户端**：微信小程序优先，方便用户随时查附近热门咖啡馆、收藏、打卡、看榜单、学咖啡知识。
* **Web 端**：用于 SEO、品牌展示、城市榜单内容沉淀、咖啡百科内容沉淀。
* **商家端**：Web 后台优先，让咖啡店老板入驻、编辑店铺资料、上传菜单、查看热度数据。
* **管理端**：平台运营人员审核商家、管理榜单、处理评论、配置推荐位。

---

## 2. 产品核心价值

### 2.1 对用户的价值

用户打开这个产品时，不应该觉得它只是一个「咖啡店列表」，而应该觉得它像一个懂咖啡、懂城市、懂口味偏好的咖啡向导。

用户能解决的问题：

1. **不知道去哪喝咖啡**：打开小程序即可看到当前城市、附近、热门商圈的咖啡馆排行。
2. **不知道哪家值得去**：通过热度、评分、收藏数、打卡数、特色标签、用户评价判断。
3. **不知道点什么咖啡**：通过咖啡百科、店铺招牌饮品、口味标签理解不同咖啡。
4. **想找特定风格咖啡馆**：比如安静办公、适合拍照、宠物友好、精品手冲、深夜营业、适合约会。
5. **想记录自己的咖啡地图**：收藏、打卡、写评价、生成个人咖啡偏好。

### 2.2 对商家的价值

商家不是被动出现在榜单里，而是可以主动入驻、经营自己的展示页。

商家能获得：

1. **品牌曝光**：出现在城市榜、商圈榜、主题榜。
2. **店铺主页**：展示图片、环境、菜单、营业时间、位置、招牌咖啡。
3. **用户反馈**：查看收藏、浏览、打卡、评论数据。
4. **运营工具**：发布活动、新品、优惠、限时推荐。
5. **数据洞察**：了解用户最关注什么、哪些咖啡最受欢迎。

### 2.3 对平台的价值

平台最终可以形成一个咖啡生活方式社区和咖啡消费入口。

长期商业化方向：

1. 商家入驻认证费。
2. 榜单推荐位。
3. 城市精品咖啡地图会员。
4. 咖啡课程、咖啡豆、器具电商。
5. 品牌联名活动。
6. 咖啡节、探店活动、城市路线。

## 3. MVP 范围定义

MVP 不要一开始做得太重，第一版目标是验证三件事：

1. 用户是否愿意用它找咖啡店。
2. 商家是否愿意入驻并维护资料。
3. 热度榜单是否能帮助用户做选择。

### 3.1 MVP 必须做的功能

#### 用户端

1. 首页推荐
2. 城市咖啡馆热度榜
3. 咖啡馆详情页
4. 搜索和筛选
5. 咖啡百科
6. 用户收藏
7. 用户打卡
8. 简单评价
9. 微信登录

#### 商家端

1. 商家注册 / 登录
2. 提交入驻申请
3. 编辑店铺信息
4. 上传店铺图片
5. 管理菜单和招牌饮品
6. 查看基础数据：浏览数、收藏数、打卡数

#### 管理端

1. 商家审核
2. 店铺信息审核
3. 评论管理
4. 城市、商圈、标签管理
5. 热度榜单人工干预

### 3.2 MVP 暂时不做的功能

这些功能可以以后做，不建议第一版就做：

1. 在线点单。
2. 外卖下单。
3. 复杂会员体系。
4. 完整社交关注流。
5. 私信聊天。
6. 商家广告投放后台。
7. 复杂 AI 推荐。
8. 自研地图导航。
9. 大规模爬虫系统。

MVP 阶段可以先用「人工录入 + 商家入驻 + 少量公开数据整理」来启动，避免一开始就被爬虫、数据版权、反爬和合规问题拖住。

## 4. 用户画像设计

### 4.1 咖啡新手

特点：

* 不懂咖啡种类。
* 不知道美式、拿铁、澳白、dirty、手冲有什么区别。
* 选择困难。

产品需要给他的体验：

* 首页直接推荐「新手友好咖啡馆」。
* 咖啡百科用简单语言解释口味。
* 点进店铺后看到「适合新手点什么」。
* 用口味标签引导：奶香、低苦、果酸、清爽、浓郁。

### 4.2 城市探店用户

特点：

* 喜欢拍照、发朋友圈/小红书。
* 关心店铺氛围、装修、出片程度。
* 愿意收藏和打卡。

产品需要给他的体验：

* 城市热门榜。
* 商圈榜。
* 高颜值咖啡馆榜。
* 店铺图片墙。
* 打卡人数、收藏人数、最近热度。

### 4.3 精品咖啡爱好者

特点：

* 关心豆子、烘焙、萃取方式。
* 关注手冲、SOE、dirty、意式机、烘豆师。
* 更看重专业信息。

产品需要给他的体验：

* 店铺支持填写豆单、烘焙程度、产区、处理法。
* 咖啡百科更专业。
* 支持筛选：手冲、烘豆店、SOE、比赛获奖咖啡师。
* 评论里支持口味维度：酸度、甜感、醇厚度、余韵。

### 4.4 商务/办公用户

特点：

* 需要找安静、稳定 Wi-Fi、有插座的咖啡馆。
* 关心座位、营业时间、人流量。

产品需要给他的体验：

* 筛选「适合办公」。
* 显示是否有 Wi-Fi、插座、安静程度。
* 显示高峰时段。
* 显示是否可长坐。

### 4.5 咖啡店商家

特点：

* 希望被用户发现。
* 希望展示品牌调性。
* 希望知道平台能不能带来客流。

产品需要给他的体验：

* 入驻流程简单。
* 后台清晰。
* 可快速编辑店铺信息。
* 可看到浏览、收藏、打卡、评价数据。
* 可申请认证，提高可信度。

## 6. 核心用户路径

### 6.1 用户找咖啡馆路径

1. 用户打开小程序。
2. 系统识别或让用户选择城市。
3. 首页展示当前城市热门咖啡馆。
4. 用户点击「热度榜」。
5. 用户按商圈、距离、评分、标签筛选。
6. 用户进入咖啡馆详情页。
7. 用户查看图片、招牌饮品、营业时间、地址、评价。
8. 用户收藏或点击导航。
9. 到店后打卡或评价。

这个路径的关键是：用户从打开到找到一家想去的店，最好在 30 秒内完成。

### 6.2 咖啡新手学习路径

1. 用户看到首页的「今天喝什么」。
2. 选择自己的口味：怕苦、喜欢奶、想提神、想尝新。
3. 系统推荐咖啡类型：拿铁、澳白、Dirty、美式、冷萃。
4. 用户进入咖啡百科。
5. 百科底部推荐「适合喝这个咖啡的店」。
6. 用户跳转到店铺详情。

这个路径可以把内容和交易/到店行为连接起来。

### 6.3 商家入驻路径

1. 商家点击「商家入驻」。
2. 填写手机号/邮箱/微信登录。
3. 提交店铺基础信息。
4. 上传营业执照或门店证明。
5. 填写营业时间、地址、联系方式。
6. 上传店铺图片和菜单。
7. 等待平台审核。
8. 审核通过后店铺上线。
9. 商家进入后台查看数据和维护信息。

MVP 阶段审核可以是人工审核，先保证内容质量。

## 7. 页面详细设计

## 7.1 用户端：首页

### 页面目标

让用户一打开就知道：这里可以快速找到好咖啡馆。

### 页面模块

1. **顶部区域**

   * 当前城市：例如「上海」
   * 定位切换
   * 搜索框：搜索咖啡馆、商圈、咖啡类型

2. **今日推荐**

   * 1 张大卡片
   * 展示当前城市最热或最适合用户偏好的咖啡馆
   * 文案示例：「今天适合去一家安静的手冲店」

3. **快捷筛选入口**

   * 附近热门
   * 高颜值
   * 适合办公
   * 精品手冲
   * 新店
   * 深夜营业

4. **城市热榜 Top 10**

   * 店铺名称
   * 排名
   * 热度值
   * 距离
   * 标签
   * 封面图

5. **咖啡百科卡片**

   * 「Dirty 和拿铁有什么区别？」
   * 「第一次喝手冲怎么选？」
   * 「美式为什么会酸？」

6. **商家入驻入口**

   * 位置不要太抢用户端体验
   * 可以放在首页底部或我的页面

### 用户体验重点

* 首页不要信息过载。
* 第一屏必须出现高质量咖啡馆卡片。
* 图片要大，标签要清楚。
* 排名不要只显示数字，要解释为什么上榜。

## 8. 热度算法 MVP 设计

MVP 阶段不要追求复杂算法，可以用透明、可解释的综合评分。

### 8.1 热度评分组成

建议热度分数满分 100：

1. 浏览热度：20%
2. 收藏热度：20%
3. 打卡热度：20%
4. 评论热度：15%
5. 评分质量：15%
6. 近期增长：10%

### 8.2 简化公式

```text
popularity_score =
  view_score * 0.20 +
  favorite_score * 0.20 +
  checkin_score * 0.20 +
  review_score * 0.15 +
  rating_score * 0.15 +
  trend_score * 0.10
```

### 8.3 排名规则

为了避免老店永远霸榜，需要做时间维度。

榜单可以分为：

1. 总热度榜：累计数据权重大。
2. 本周热榜：最近 7 天数据权重大。
3. 新店榜：开业 90 天内或平台收录 30 天内。
4. 商圈榜：只在特定区域内排名。

### 8.4 反作弊策略

MVP 阶段先做基础反作弊：

1. 同一用户短时间多次浏览只计一次有效浏览。
2. 同一用户只能收藏一次。
3. 同一用户每天同一家店只允许一次打卡。
4. 评论需要登录。
5. 异常增长进入人工审核。
6. 商家不能给自己的店刷评论。

## 9. 推荐系统 MVP 设计

MVP 不需要复杂机器学习，先做规则推荐。

### 9.1 用户偏好标签

用户可以选择：

* 喜欢奶咖
* 喜欢黑咖
* 怕苦

## 10. 技术架构建议

## 10.1 总体架构

建议采用：

```text
用户小程序 / Web 前台 / 商家后台 / 管理后台
              ↓
        API Gateway / FastAPI
              ↓
     PostgreSQL + Redis + 对象存储
              ↓
      后台任务 / 数据采集 / 排名计算
```

### 前端

1. **Web 前台**：Next.js + React + TypeScript
2. **商家后台**：Next.js + React + TypeScript
3. **管理后台**：Next.js + React + TypeScript
4. **微信小程序**：建议用 Taro + React，或原生微信小程序

### 后端

1. Python FastAPI
2. SQLAlchemy 或 SQLModel
3. Pydantic 数据校验
4. Alembic 数据库迁移
5. JWT / 微信登录 Session
6. Redis 缓存
7. Celery / RQ / APScheduler 后台任务

### 数据库

MVP 推荐：

1. PostgreSQL：主数据库
2. Redis：缓存、排行榜、验证码、限流
3. S3 / COS / OSS：图片存储

为什么不用一开始就上复杂数据库：

* 咖啡馆、用户、评论、收藏、打卡都是典型关系型数据。
* PostgreSQL 足够支撑 MVP 和中期增长。
* 后续可以加 Elasticsearch / Meilisearch 做更强搜索，但 MVP 可以先用 PostgreSQL 搜索能力。

## 17. 合规和风险控制

### 17.1 数据采集风险

如果从外卖平台、点评平台等采集数据，需要非常谨慎。

风险包括：

1. 平台服务条款限制。
2. 数据版权问题。
3. 用户评论版权问题。
4. 商家信息准确性问题。
5. 反爬机制和账号封禁风险。

### 17.2 更安全的 MVP 做法

1. 不直接复制第三方评论。
2. 不直接搬运第三方图片。
3. 优先使用商家授权图片。
4. 店铺基础信息可通过商家提交和人工核验。
5. 用户评价只使用平台内生成内容。
6. 对商家提供认领和纠错入口。

### 17.3 用户隐私

需要注意：

1. 微信登录授权说明清晰。
2. 地理位置授权必须由用户主动同意。
3. 用户可以删除评价或打卡。
4. 后台不要暴露用户敏感信息。
5. 日志中不要记录过多个人隐私。

## 24. 结论

这个项目的核心不是简单展示咖啡馆，而是建立一个「以咖啡热度和用户偏好为核心的城市咖啡发现平台」。

第一版不要追求大而全，应该聚焦：

1. 用户能快速找到值得去的咖啡馆。
2. 商家能顺利入驻并维护信息。
3. 平台能生成可信、可解释的热度榜单。
4. Web 和小程序分别承担不同角色：小程序负责高频使用，Web 负责搜索流量、内容沉淀和后台管理。

只要 MVP 能把这四件事做好，后续就可以自然扩展到社区、活动、商业化和咖啡电商。
